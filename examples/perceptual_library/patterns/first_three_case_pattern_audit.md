# First Three FLOC* Perceptual Case Pattern Audit

Generated: 2026-05-15
Scope: cross-case audit only
Cases: Charms, D4DATA, Grandvalira Resorts
Schema: `examples/perceptual_library/schema/perceptual_case_record_schema.json`

## Purpose

This audit compares the first three `perceptual_case_record` fixtures to identify early Brand3/FLOC* perceptual grammar. It does not modify the schema, case records, scoring, reports, runtime, or Visual Signature.

The three cases intentionally pressure different surfaces:

- Charms: editorial / minimal / atmospheric.
- D4DATA: experimental / motion-heavy / metaphor-led.
- Grandvalira Resorts: corporate / design-system / multi-brand.

## What Is Emerging

The strongest shared grammar is not a shared visual style. It is a shared method: FLOC* case language repeatedly converts abstract categories into perceptual systems that can be read as behavior.

Across the first three cases, Brand3 should look for:

- a conceptual source that organizes the surface
- a system behavior that makes the concept legible
- a tension between stated category and surface expression
- a review boundary where copy claims exceed observed evidence

## Differences Between Cases

### Charms

Charms is the strongest visual-observation fixture because it has prior local visual artifacts. Its grammar is atmospheric and editorial: softness, ritual, silence, recurring forms, threshold-like interaction.

Schema fit: strong. The record can use high-confidence visual observations because they are anchored to local prior case artifacts.

Primary risk: overgeneralizing Charms' softness as a Brand3/FLOC* default.

### D4DATA

D4DATA pressure-tests motion and metaphor. It has precise public language about kendo, Datamosh, MetaHuman, transitions, intensity, and precision, but first-pass evidence is mostly copy.

Schema fit: workable but frictional. The schema can represent the case, but visual observations become `copy_based` and must stay medium or low confidence.

Primary risk: treating production-method copy as observed motion.

### Grandvalira Resorts

Grandvalira pressure-tests corporate system language. It is about multi-brand structure, components, foundations, scalability, visual cohesion, and unique personalities.

Schema fit: workable but incomplete. The schema can represent the claims, but it lacks variant/sub-brand evidence structure.

Primary risk: treating design-system claims as validated system performance without component, token, or multi-screen evidence.

## Where The Schema Works

- It forces facts, observations, interpretations, weak inferences, and contradictions into separate layers.
- Evidence anchoring prevents copy claims from silently becoming visual truth.
- Confidence semantics keep copy-only interpretations out of high confidence.
- `requires_human_review` is doing useful containment work.
- Contradictions work well for productive tensions, not only failures.
- The same schema can hold atmospheric, motion-heavy, and system-heavy cases without immediate structural changes.

## Where The Schema Creates Friction

The friction log points to consistent issues:

- `visual_observations` currently holds both observed visual behavior and stated visual behavior.
- Motion evidence needs a more precise evidence medium: static capture, video, scroll session, production-method copy.
- Conceptual metaphor may deserve its own layer separate from visual observations.
- Contradictions need basis metadata: copy-only, capture-only, copy-vs-capture, mixed.
- Human review needs per-item resolution states, not only a top-level status.
- Multi-brand system records need variant/sub-brand evidence.
- Corporate system cases need evidence standards for scalability, cohesion, foundations, and components.

The schema should not be changed from this audit alone, but these issues should shape the next schema revision.

## Future Report Prompt Implications

Future report prompts should learn the method, not the case voice:

- Ask for fact/observation/interpretation separation.
- Require evidence anchors for every perceptual claim.
- Require confidence semantics in prose: direct, copy-based, inferred, weak.
- Ask for productive tensions, not only strengths and weaknesses.
- Prevent copy-only claims from becoming confirmed visual conclusions.
- Encourage phrases like `stated by the source`, `visible in the surface`, `inferred from copy`, and `requires review`.
- Add a specific warning for system claims: scalability, cohesion, modularity, and components need multi-surface evidence.

Prompt language should avoid importing slogans or metaphors directly from FLOC* cases unless discussing those exact cases.

## Visual Signature Review Implications

Future Visual Signature review questions should include:

- What is directly visible versus only stated in copy?
- Which repeated signals create a system rather than decoration?
- Is the concept visible in hierarchy, motion, components, or interaction?
- Does motion reinforce a concept, or is motion only claimed?
- Does a design-system claim show components, variants, tokens, or multi-screen consistency?
- Where does the surface preserve difference while creating cohesion?
- What productive contradiction appears between stated claim and observed signal?
- Which low-confidence inferences need human review before reuse?

## Audit Conclusion

The first three cases suggest that Brand3/FLOC* perceptual grammar should center on evidence-bound translation: abstract domain language becomes readable through surface behavior, motion behavior, or system behavior. The most reusable layer is not any one style. It is the disciplined distinction between what the source states, what the surface shows, what Brand3 infers, and what still needs human review.
