# Original Brand3 Initial Scoring Flow Audit

Generated: 2026-05-13
Scope: audit only

This audit documents the Brand3 initial scoring flow as it exists in the repo before using it as an input to a unified Brand3 Platform. It does not modify scoring logic, rubric dimensions, Visual Signature, capture behavior, or production/local reports.

## Executive Summary

Brand3 already has a complete initial scoring path that is separate from Visual Signature:

- URL/brand input enters through the local FastAPI web form or the CLI.
- The analysis service collects raw inputs, extracts features, scores dimensions, writes SQLite records, and emits a per-run JSON file under `output/`.
- Dimension scores are stored in both SQLite and JSON outputs.
- Full report prose is generated by the report renderer from SQLite snapshots; a dedicated persisted `generated_texts_per_dimension` artifact was not found.
- The existing scoring UI/viewer is the FastAPI + Jinja app and the HTML report renderer.

Visual Signature must remain a separate evidence/review layer. It must not feed scoring, change rubric dimensions, alter composite score, or change production/local report behavior.

## 1. How The Project Accepted A New Brand Or URL

### Web Form

Found.

Files:

- `web/routes/index.py`
- `web/templates/index.html.j2`
- `web/routes/analyze.py`
- `web/workers/url_validator.py`
- `web/storage.py`
- `migrations/005_web_requests.sql`

Flow:

1. `GET /` renders a form with a single `url` input.
2. The form posts to `POST /analyze`.
3. `web/routes/analyze.py` validates the URL.
4. It creates a public token and brand slug.
5. It inserts a row in `web_requests`.
6. It enqueues the token in `web/workers/queue.py`.
7. It redirects to `/r/{token}/status`.

### CLI

Found.

Primary command:

```bash
python main.py analyze <url> [brand_name]
```

Legacy shorthand:

```bash
python main.py <url> [brand_name]
```

`main.py` normalizes a URL-first invocation into `analyze <url>`.

Related CLI command:

```bash
python main.py enqueue-job <url> [brand_name]
```

Found.

Per-run JSON files are written to:


## 4. Where Generated Texts Per Dimension Are Stored

Partially found.

There are several text-like layers, but not a single stored `generated_texts_per_dimension` artifact.

### Stored Summary

Location:

- SQLite `runs.summary`

Source:

- `ScoringEngine.generate_summary(...)`

This is a global summary with per-dimension score lines, weakest dimension, and strongest dimension. It is not rich per-dimension prose.

### Stored Dimension Observations

Location:

- SQLite `scores.insights_json`
- SQLite `scores.rules_json`

Source:

- `DimensionScore.insights`
- `DimensionScore.rules_applied`

These are rule/insight lists and are often empty. They are not full report prose.

### Report Prose / Findings

Derived at render time from SQLite snapshots.

Files:

- `src/reports/renderer.py`
- `src/reports/derivation.py`
- `src/reports/dossier.py`
- `src/reports/narrative.py`
- `src/reports/templates/report.html.j2`

Source data:

- `runs`
- `scores`
- `features`
- `evidence_items`
- `raw_inputs`

`src/reports/derivation.py` builds per-dimension report blocks from scores, features, evidence, confidence, and readiness. `src/reports/dossier.py` can apply narrative overlays through `src/reports/narrative.py`, with deterministic fallback behavior.

Status:

```text
missing_or_unknown: dedicated persisted generated_texts_per_dimension artifact
```

Recommended SQLite tables:

- `brands`
- `runs`
- `raw_inputs`
- `features`
- `scores`
- `evidence_items`
- `run_audits`
- `web_requests`
- `analysis_jobs`
- `analysis_job_events`
- `llm_cache`

Avoid:

- Do not call `brand_service.run(...)` from a static platform builder.
- Do not invoke collectors.
- Do not invoke providers.
- Do not recompute scores.
- Do not mutate SQLite.
- Do not render reports if that would trigger provider-backed narrative work.

## 8. What Must Remain Separate From Visual Signature

The unified platform may summarize and link to both layers, but these must remain separate:

- Initial scoring dimensions and `composite_score`
- `src/dimensions.py`
- `src/scoring/engine.py`
- `src/services/scoring_pipeline.py`
- SQLite scoring tables: `runs`, `features`, `scores`
- Scoring report renderer and templates
- Visual Signature captures, review queues, calibration, governance, corpus expansion
- Visual Signature evidence persistence and shadow-run behavior

Boundary:

```text
Visual Signature must not feed Initial Scoring, alter dimension_scores, alter rubric dimensions, alter report copy, or enable runtime mutation.
```

## Build Implications For The Unified Platform

The platform should treat scoring as an existing read-only layer:

- Prefer `data/brand3.sqlite3` for authoritative run history, features, scores, summaries, and report-ready snapshots.
- Use `output/*.json` as file-based run exports and fallback discovery.
- Use `output/reports/**/*.html` as existing rendered report links.
- Use `src/dimensions.py` only for displaying rubric definitions, not modifying them.
- Keep Visual Signature artifacts under their existing `examples/visual_signature/...` roots and present them in separate sections.
